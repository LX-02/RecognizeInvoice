# AGENTS.md

本文件是 Codex 在本项目每个 session 开始时应优先阅读的项目说明。它用于保持开发方向一致，避免第一版实现范围失控。

## 项目目标

本项目用于建设一个面向企业 ERP/报销场景的发票自动识别系统。

核心目标：

- 用户上传发票图片或 PDF。
- 系统自动识别发票中的关键字段。
- 财务或报销人员可以查看、确认、修正识别结果。
- 识别后的结构化数据可用于报销审核、入账、重复校验和后续 ERP 流程。

## 第一版 MVP 范围

MVP 指 Minimum Viable Product，即“最小可用产品”。本项目第一版只做必要内容的提取和确认，不追求一次性覆盖所有票种、所有税务场景和所有自动化审核规则。

第一版重点是：

- 关键字段能识别。
- 识别结果结构清楚。
- 用户可以人工确认和修改。
- 能做基础重复校验。
- 数据能进入后续报销/ERP 流程。

## 第一版必须提取的字段

### 发票身份

- 发票类型
- 发票代码
- 发票号码
- 数电票号码
- 开票日期
- 校验码
- 二维码内容或二维码识别结果

### 购买方信息

- 购买方名称
- 购买方纳税人识别号/统一社会信用代码

### 销售方信息

- 销售方名称
- 销售方纳税人识别号/统一社会信用代码

### 金额和税额

- 不含税金额
- 税率/征收率
- 税额
- 价税合计小写
- 价税合计大写

### 发票明细

- 项目名称/商品服务名称
- 规格型号
- 单位
- 数量
- 单价
- 金额
- 税率
- 税额

第一版可以先支持主要明细行识别；若多行明细解析成本较高，可以先保证合计金额、税额和主要项目名称准确。

### 辅助信息

- 备注
- 开票人
- 原始文件路径
- 原始图片/PDF 文件哈希值
- OCR 原文
- 字段识别置信度

## 第一版建议具备的基础校验

- 发票号码/代码/数电票号码重复校验。
- 发票金额是否为空或异常。
- 开票日期是否为空或格式异常。
- 购买方名称或税号是否与当前企业信息匹配。
- OCR 低置信度字段标记为需要人工确认。

## 暂不作为第一版重点的内容

以下功能可以作为第二版或后续版本规划，不要在第一版中主动扩大范围，除非用户明确要求：

- 发票真伪在线查验。
- 进项税勾选认证。
- 合同、订单、付款单自动匹配。
- 复杂费用报销审批流。
- 风险模型、供应商黑名单、异常税率分析。
- 覆盖所有特殊票据类型，如机票、火车票、出租车票、机动车发票等。
- 全自动凭证生成。

## 数据设计原则

识别结果应区分原始值、标准化值和人工修正值。

建议字段模型保留：

- `raw_value`: OCR 或模型直接识别出的原始文本。
- `normalized_value`: 标准化后的值，例如日期、金额、税号。
- `confidence`: 字段置信度。
- `source_bbox`: 字段在原图/PDF 中的位置，若识别引擎支持。
- `manual_value`: 用户人工修正后的值。
- `validation_status`: 校验状态。

## 开发原则

- 优先满足第一版 MVP，不要过早抽象。
- 字段命名要稳定、清晰，便于后续接入 ERP。
- 对金额、日期、税号等关键字段要做标准化。
- 保留原始文件和 OCR 原文，方便追溯。
- 对低置信度或校验失败字段，应进入人工确认流程。
- 不要静默丢弃识别失败的信息，应明确记录失败原因。

## 协作约定

- 如果需求不明确，优先围绕“企业报销发票识别第一版 MVP”做保守实现。
- 修改代码前先阅读现有项目结构和已有约定。
- 不要删除用户已有改动。
- 新增功能时优先保持简单可运行，再逐步增强。
- 重要字段、接口或数据结构变更，应同步更新文档。
